## 一、项目背景与目标

### 1.1 项目背景

在无网络覆盖的环境下（如户外、地下空间、应急场景等），传统即时通信应用无法提供服务。本项目基于鸿蒙系统的 **Wi-Fi Direct（P2P）** 技术，构建一套不依赖任何网络基础设施和账号体系的跨设备即时通信系统。

Wi-Fi Direct 允许设备在没有传统网络设备（如路由器）的情况下直接通过 Wi-Fi 进行通信。在此技术基础上，结合 Socket 编程和自定义应用层协议，实现设备间的文字消息和图片内容传输。

### 1.2 项目目标

构建一款基于鸿蒙系统的轻量级跨设备即时通信应用，满足以下核心诉求：

- **零基础设施依赖**：无需 Wi-Fi 热点、路由器、蜂窝网络或互联网接入
- **零账号体系**：无需注册、登录任何账号，设备即身份
- **即时通信**：支持文字、图片的实时发送与接收
- **多模式聊天**：支持单聊（一对一）和群聊（一对多）两种模式
- **跨设备兼容**：在支持 Wi-Fi Direct 的鸿蒙设备间实现互联互通


## 二、技术架构方案

### 2.1 整体架构

系统采用分层架构设计，自上而下分为：

| 层级 | 模块 | 说明 |
|------|------|------|
| **应用层** | UI界面、会话管理、消息展示 | 用户交互与界面呈现 |
| **协议层** | 自定义应用层协议 | 消息封装、解析、路由 |
| **传输层** | TCP Socket | 可靠数据传输 |
| **网络层** | Wi-Fi Direct (P2P) | 设备发现、组网建立 |
| **物理层** | Wi-Fi 硬件 | 2.4GHz/5GHz 无线通信 |

### 2.2 核心组件

#### 2.2.1 设备发现与组网模块

基于 `@ohos.wifiManager` 提供的 P2P 接口实现：

- **设备发现**：通过 `startDiscoverDevices()` 扫描附近设备
- **设备监听**：通过 `on('p2pPeerDeviceChange')` 监听设备状态变化
- **群组创建**：通过 `createGroup()` 或 `p2pConnect()` 建立 P2P 连接
- **组网模式**：
  - **临时组（netId: -1）** ：每次连接重新协商
  - **永久组（netId: -2）** ：保存群组信息，下次快速连接

#### 2.2.2 数据传输模块

基于 `@kit.NetworkKit` 的 Socket API 实现：

- **服务端**：Group Owner（GO）启动 TCP ServerSocket，监听客户端连接
- **客户端**：Group Client（GC）通过 TCP Socket 连接 GO
- **数据传输**：通过 Socket 的 `send()` 和 `on('message')` 进行双向通信

#### 2.2.3 协议解析模块

自定义应用层协议，负责消息的编解码、路由和分发。

### 2.3 网络拓扑

系统采用 **星型拓扑结构**：

- **Group Owner (GO)**：群组中心节点，负责管理群组成员和消息转发
- **Group Client (GC)**：群组成员节点，通过 GO 与其他成员通信

```
                    ┌─────────────┐
                    │   GO 设备   │
                    │ (群组所有者) │
                    └──────┬──────┘
                           │
           ┌───────────────┼───────────────┐
           │               │               │
      ┌────┴────┐    ┌────┴────┐    ┌────┴────┐
      │ GC 设备 │    │ GC 设备 │    │ GC 设备 │
      │ (成员A) │    │ (成员B) │    │ (成员C) │
      └─────────┘    └─────────┘    └─────────┘
```

## 四、非功能需求

### 4.1 性能需求

| 指标 | 要求 | 说明 |
|------|------|------|
| 设备发现时间 | ≤ 3 秒 | 扫描并显示附近设备 |
| 连接建立时间 | ≤ 5 秒 | 从发起连接到群组建立完成 |
| 消息发送延迟 | ≤ 200ms | 文字消息从发送到接收的端到端延迟 |
| 图片传输速度 | ≥ 10Mbps | 基于 Wi-Fi Direct 的高速传输能力 |
| 并发连接数 | ≥ 8 台 | 单个群组支持的最大设备数 |
| 消息吞吐量 | ≥ 100 条/秒 | 群组内消息并发处理能力 |

### 4.2 可靠性需求

| 需求 | 说明 |
|------|------|
| 断线重连 | 连接异常断开后自动尝试重连，重连间隔 3 秒，最多尝试 5 次 |
| 消息可靠性 | TCP 协议保证消息可靠送达，不丢失、不重复 |
| 心跳机制 | 每 10 秒发送心跳包检测连接状态，连续 3 次无响应判定为离线 |
| 异常恢复 | 应用异常退出后重新进入时自动恢复连接 |

### 4.3 安全需求

| 需求 | 说明 |
|------|------|
| 连接授权 | 设备连接需经双方手动确认，防止未授权接入 |
| 数据加密 | 敏感数据（如图片）传输建议使用 TLS Socket 加密 |
| 本地隔离 | 消息数据仅存储在本地设备，不上传任何云端服务器 |

### 4.4 兼容性需求

| 需求 | 说明 |
|------|------|
| 系统版本 | 支持 HarmonyOS 5.0.0 (API 12) 及以上版本 |
| 硬件要求 | 设备需支持 Wi-Fi Direct 功能（SystemCapability.Communication.WiFi.P2P） |
| 设备类型 | 支持手机、平板、智慧屏等鸿蒙设备 |

### 4.5 易用性需求

| 需求 | 说明 |
|------|------|
| 零配置 | 无需任何网络配置，打开应用即可使用 |
| 零注册 | 无需注册账号，设备即身份 |
| 操作直观 | 设备发现、连接、聊天等核心操作路径清晰简洁 |
| 状态可视 | 设备在线/离线状态、消息发送状态明确可见 |


## 五、应用层协议设计

### 5.1 协议基础

所有消息采用 **JSON 格式** 作为消息体结构，通过 TCP Socket 进行传输。每条消息以 `\n` 作为消息结束符。

### 5.2 通用消息结构

```json
{
  "version": "1.0",
  "type": "MESSAGE_TYPE",
  "from": "sender_device_id",
  "fromName": "发送者设备名称",
  "to": "receiver_device_id",
  "timestamp": 1700000000000,
  "payload": { ... }
}
```

### 5.3 消息类型定义

| 类型 | 说明 | 方向 |
|------|------|------|
| `TEXT` | 文本消息 | 双向 |
| `IMAGE` | 图片消息 | 双向 |
| `SYSTEM_JOIN` | 成员加入通知 | GO → 所有成员 |
| `SYSTEM_LEAVE` | 成员离开通知 | GO → 所有成员 |
| `HEARTBEAT` | 心跳包 | 双向 |
| `ACK` | 消息送达确认 | 双向 |
| `FILE_REQUEST` | 请求传输文件 | 双向 |
| `FILE_DATA` | 文件数据分片 | 双向 |

### 5.4 消息示例

**文本消息**：
```json
{
  "version": "1.0",
  "type": "TEXT",
  "from": "device_123456",
  "fromName": "张三的手机",
  "to": "device_789012",
  "timestamp": 1700000000000,
  "payload": {
    "content": "你好，这是一条测试消息"
  }
}
```

**图片消息**：
```json
{
  "version": "1.0",
  "type": "IMAGE",
  "from": "device_123456",
  "fromName": "张三的手机",
  "to": "device_789012",
  "timestamp": 1700000000000,
  "payload": {
    "fileName": "photo_20260624.jpg",
    "fileSize": 2048576,
    "fileMd5": "d41d8cd98f00b204e9800998ecf8427e",
    "totalChunks": 250,
    "chunkSize": 8192
  }
}
```
> 图片消息发送时，先发送此 JSON 元数据，随后通过 Socket 连续发送二进制图片数据分片。

**群聊消息**（`to` 字段为 `"GROUP"`）：
```json
{
  "version": "1.0",
  "type": "TEXT",
  "from": "device_123456",
  "fromName": "张三的手机",
  "to": "GROUP",
  "timestamp": 1700000000000,
  "payload": {
    "content": "大家晚上好！"
  }
}
```


## 六、界面原型需求

### 6.1 主要页面

| 页面 | 功能 |
|------|------|
| **设备发现页** | 显示本机设备信息、扫描附近设备列表、连接/断开操作 |
| **会话列表页** | 展示所有历史会话（单聊+群聊），按时间排序 |
| **单聊会话页** | 一对一聊天界面，展示消息列表、输入框、发送按钮 |
| **群聊会话页** | 群组聊天界面，展示消息列表（含发送者名称）、输入框、发送按钮 |
| **图片预览页** | 点击图片消息后全屏查看大图 |

### 6.2 交互要求

- 设备列表中的在线设备与离线设备视觉区分
- 聊天消息按时间顺序展示，自己发送的消息靠右，接收的消息靠左
- 图片消息以缩略图形式展示，点击可查看大图
- 消息发送中显示 loading 状态，发送失败显示重发按钮
- 新消息到达时，如当前不在该会话页面，会话列表显示未读角标


## 七、约束与限制

### 7.1 技术约束

- 必须基于 HarmonyOS 原生开发（ArkTS），不得使用跨平台框架
- Wi-Fi Direct 相关操作必须通过 `@ohos.wifiManager` 系统 API 实现
- Socket 通信必须通过 `@kit.NetworkKit` 系统 API 实现
- 应用权限需在 `module.json5` 中声明

### 7.2 运行约束

- 所有参与通信的设备必须开启 Wi-Fi 功能
- 设备间通信距离受 Wi-Fi 信号强度影响，建议在有效覆盖范围内使用
- 设备数量增加时，GO 设备的 CPU 和内存负载会相应增加

### 7.3 功能限制

- 不支持离线消息（设备离线期间发送的消息无法接收）
- 不支持跨子网通信（所有设备必须在同一 P2P 群组内）
- 不支持音视频通话（本版本暂不实现）
